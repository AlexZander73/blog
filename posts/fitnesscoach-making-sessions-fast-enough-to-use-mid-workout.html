FitnessCoach: Making Sessions Fast Enough to Use Mid-Workout

I focused on in-session speed because the best workout plan still fails if logging is annoying. The active session and recap flow became the practical center of daily use.

March 23, 20264 min read

Apps Design Build Logs

Where theory meets friction

The most practical work in this phase was session flow. I added an active session mode with quick presets, set logging, rest timing, swaps, and recap surfaces. I remember feeling protective of this part because it is where intent gets tested against real-world friction. If this flow is clumsy, everything else in the app becomes theory.

Building the start-log-adjust-finish loop

The concrete change landed in the new session pages and store actions. I built the "start, log, adjust, finish, recap" path so I could complete a full session inside the app without context-switching. Quick set presets and swap flows were especially important because those are the moments where people abandon apps when interactions take too many taps.

I also kept explanation surfaces close to action. If an exercise is swapped, the user should see why in plain language, not hunt through hidden logic. That aligns with the deterministic engine goal and keeps trust high during adjustment-heavy days.

Speed has an emotional effect

The lesson was that speed is emotional, not just technical. When logging felt immediate, I felt calmer using the app during a session. When it felt heavy, I avoided it. That is not a performance benchmark; it is behavior reality.

This work also exposed a follow-on need: richer interaction instrumentation. I can design for low friction, but I still need actual completion and abandonment data to confirm where the flow holds up and where it breaks.

Next move: instrument session funnel events so I can measure exactly where users drop off between start, first set, and recap save.
