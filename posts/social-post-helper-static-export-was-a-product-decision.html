Social Post Helper: Static Export Was a Product Decision

I treated deployment as part of product scope, not something to bolt on later. Shipping this as a static export pushed the app toward a cleaner and more disciplined MVP.

March 12, 20262 min read

Apps Systems Build Logs

Let deployment shape the product

I like it when deployment stops being interesting. For Social Post Helper, that meant I did not want a server if I could avoid it. The whole product premise was already pointing that way: media metadata extraction in the browser, local draft generation, local persistence, and no auth. Once I saw that shape clearly, the hosting choice stopped feeling separate from the product. I felt relief, honestly. A static app meant I could protect the scope instead of arguing with it.

A real release path from day one

The concrete change was setting the app up for static export and GitHub Pages from the start. The Next.js config uses export mode, keeps images unoptimized for static hosting, and handles a base path so the app can live at a project URL. The repository also includes a deployment workflow that installs dependencies, runs tests, builds the static output, and pushes the Pages artifact. That matters because it turns "it should probably deploy somewhere" into a real release path.

This choice narrowed other decisions in a healthy way. If there is no server, then backend-heavy feature drift becomes harder to justify. Metadata parsing has to stay browser-safe. Persistence has to stay local. Reverse geocoding has to be optional and resilient. Even future AI integration needs a clean boundary instead of bleeding into every module. I like what that pressure did to the architecture.

Scope control, not just hosting

The lesson was that infrastructure can either blur a product or sharpen it. In this case, static export sharpened it. It made me ask better questions: what truly belongs in the browser, what can fail gracefully, and what does the MVP actually need to be valuable? That is a much better design conversation than "what stack do I want to maintain?"

There is still a cost. Static export means I do not get server-side enrichment for free, and anything smarter later will need care. But that is a fair price for a first version that stays honest and easy to ship.

Next move: verify how stable the GitHub Pages path is in real use and keep the deployment boring.
