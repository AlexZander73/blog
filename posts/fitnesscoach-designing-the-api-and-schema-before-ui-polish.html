FitnessCoach: Designing the API and Schema Before Chasing UI Polish

I chose to stabilize contracts and schema early, before polishing every client interaction. It slowed visible UI progress for a bit, but it prevented cross-layer drift.

March 20, 20264 min read

Systems Build Logs Notes

Choosing invisible work first

On 2026-03-12, I made a decision that felt boring but saved me later: lock API contracts and schema structure before chasing visual polish. The repository shows this clearly in the same period that introduced docs/api-contract.md, docs/database-schema.md, and the Prisma schema. I remember feeling impatient because interface work is immediately rewarding, while contract work is mostly preventive.

Versioned contracts and a stable target

The concrete change was versioned API design under /api/v1, with an explicit envelope and shared Zod contracts in packages/contracts. I also mapped core domains in Prisma across identity, planning, sessions, symptoms, safety, metrics, and educational content. Once that model existed, client and admin surfaces had a more stable target.

The tradeoff was speed perception. Early product demos look better with shiny screens than with well-typed contracts nobody sees. But I had already felt this pain in past projects: once UI grows around unstable models, every fix becomes expensive. I did not want to relive that.

Schema design is product design

The key lesson was that schema design is product design in disguise. If symptom reports are unstructured, safety logic gets weak. If workout logs are shallow, analytics become noise. If endpoint versioning is absent, future changes become breaking changes. That is not backend purity; it directly affects user trust.

By handling these system boundaries early, I got a cleaner runway for later UX sprints. When I added deeper pages and session workflows in the next major update, I was wiring into known contracts rather than improvising new payloads under pressure.

Next move: finish auth integration with explicit authorization boundaries so these contracts can safely support multi-user production use.
