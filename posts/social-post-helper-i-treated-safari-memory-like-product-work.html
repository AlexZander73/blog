Social Post Helper: I Treated Safari Memory Like Product Work

Media apps earn suspicion fast when a browser tab starts feeling heavy. I decided that if Safari memory pressure was even a plausible concern, the product needed a direct answer instead of a shrug.

March 17, 20263 min read

Apps Systems Build Logs

A suspicious tab is a product problem

The most recent shift in Social Post Helper was not about new drafting output. It was about trust. When a media-heavy browser tab starts looking expensive in Safari, users do not care whether the leak is technically "their fault" or "the browser's fault." They just remember that your tab felt risky. I felt defensive for about five seconds when this came up, and then I realized that reaction was useless. If the product touches uploaded media and previews, memory behavior is part of the product.

Adding visible cleanup behavior

So I made concrete changes in the current workspace. I tightened cleanup around temporary image and video decoder objects, made sure preview URLs get released more aggressively, added a small RAM summary panel to the detected-info view, and built a 5-minute inactivity pause for unfocused tabs. That pause stops draft activity, releases previews, and makes resuming explicit. I also checked the surrounding behavior so the pause state does not race with ongoing uploads and so the current code still passes tests, lint, and type checks.

What I like about this work is that it turns a vague fear into visible behavior. A user can now see the app acknowledge resource use, and the app actively backs off when it sits idle in the background. That feels more honest than assuming garbage collection timing will save me. With media features, object URLs and decoder cleanup are not trivia. They are part of whether the product feels careful.

Performance work has an emotional side

The lesson here was that performance and memory are emotional problems before they are technical ones. A tab that feels suspicious loses trust quickly. I would rather slightly over-manage cleanup than leave users wondering whether the app is quietly hanging onto their files. The feature is modest, but the posture matters.

This is also a reminder that "lightweight MVP" is not an excuse. If I want to claim the app is simple, then it has to behave like a simple tool under stress too.

Next move: measure the effect of the inactivity pause in Safari's memory tools and decide whether the app should become even more aggressive about releasing media state.
