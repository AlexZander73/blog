Barbell Easy Calculator: Why I Chose React + Capacitor for a Gym Utility

I chose React + Capacitor because I needed fast iteration and mobile delivery from one codebase. The decision was practical, not ideological, and it came with tradeoffs I accepted up front.

March 20, 20264 min read

Systems Apps Tools

What I needed from the stack

When I scoped this project, I cared about one thing more than framework debates: could I ship a reliable tool quickly for iPhone and Android without splitting my momentum? That pushed me toward React, Capacitor, and TypeScript. The emotional moment was honestly relief. Once I saw the project structure running end to end, I knew I had a path I could sustain instead of a pile of parallel tracks.

The concrete change was setting up a single app surface with clear scripts for building, syncing, and opening native projects. I put in the Capacitor setup, iOS project files, and a workflow that kept the app runnable as I changed product behavior. The repository now reflects that choice everywhere: package dependencies, scripts, and native scaffolding all align with one shared codebase.

The tradeoff was worth it

The tradeoff is obvious. Web-to-native layering can feel awkward at times, and platform details still leak through. I accepted that because this app is primarily a utility with deterministic logic and straightforward interaction patterns. I do not need heavy native-only rendering tricks to deliver value. I need speed, correctness, and maintainable iteration.

That decision also kept the project emotionally lighter. One shared codebase meant fewer identity splits between “the app” and “the mobile port.” It stayed one thing.

What I learned

The lesson I took from this decision is that architecture should serve usage context, not ego. In a gym tool, consistency and fast updates matter more than perfect theoretical purity. If a user can open the app quickly, set a target, and load the bar with confidence, the stack has done its job.

I am keeping this choice honest. If future requirements push deeply into native-only constraints, I should revisit it. But for this MVP scope, React plus Capacitor was the right compromise.

Next move: validate Android device behavior on real mid-range hardware and document any platform-specific adjustments needed.
