Backlog Entry: Why I Removed Pages and Went Native-First

Backlog entry for 2026-03-10. This was a direction lock. I removed the Pages deploy path from the app repo and aligned docs/tooling around native release work.

March 10, 20264 min read

Systems Apps Notes

Why the pivot happened

The app started with web-friendly assumptions. In practice, target users kept pointing to phone-first usage. Maintaining equal focus on Pages and native created extra admin work without enough user benefit.

So I made the call: native-first for the product repo. Web still matters, but as support and marketing surface on the main site, not as the primary runtime channel.

What changed immediately

I removed deployment pieces that no longer matched the strategy and tightened release docs around App Store flow. That sounds small, but fewer parallel pipelines means fewer hidden errors.

This choice also reduced decision fatigue. When channel strategy is fuzzy, every task feels heavier than it should.

Lesson and next move

Lesson: one clear distribution path beats two half-maintained paths.

Next move was to finish privacy and metadata readiness, then push through TestFlight and IAP validation with a tighter checklist.

Next move: close metadata and billing state gaps without reopening channel drift.
