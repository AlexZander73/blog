Social Post Helper: HEIC Support Is Not One Format

I learned the usual lesson about file formats the annoying way. Saying I support HEIC means very little if the browser and picker disagree about what that format looks like.

March 13, 20262 min read

Apps Notes Build Logs

The claim was cleaner than reality

I thought HEIC support was going to be a straightforward checkbox. The product already knew it needed to handle common photo formats, and HEIC was obviously important because of Apple devices. Then the real-world edges showed up. Some files came through as different extensions. Some came through with MIME variants I had not accounted for. I felt that familiar mix of embarrassment and irritation that comes from realizing the product claim was cleaner than the actual implementation.

Expanding the actual support surface

The concrete fix was expanding support beyond a narrow "HEIC" label. I added more HEIC and HEIF extension variants and broader MIME coverage, including sequence-style variants that browsers can surface differently. Just as important, I added a support test so this would stop being an assumption and start being something the repo actually checks. That test is a quiet milestone. It means the project now records what "supported" is supposed to mean.

What changed for me was not just the format list. It was the way I thought about compatibility. A format is not only a parser problem. It is also a browser problem, a file picker problem, and sometimes a platform labeling problem. If a user cannot even select the file cleanly, it does not matter how good the metadata extraction is after that.

Backing support with tests

The lesson was blunt: compatibility work needs to be grounded in observed variants, not simplified labels. I do not get to say "HEIC works" because I added one extension and moved on. I get to say it only if the app handles the messy version of reality users actually bring with them.

This was one of those small fixes that changed the tone of the project. It made me more cautious about broad support claims and more willing to back them with targeted tests instead of confidence.

Next move: keep collecting real file variant failures so the support matrix reflects what people actually upload, not what I assume they upload.
