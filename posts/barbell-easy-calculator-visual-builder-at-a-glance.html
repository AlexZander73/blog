Barbell Easy Calculator: Designing the Visual Builder for At-a-Glance Use

I built the visual builder because some users think in stacks, not equations. The best interaction pattern turned out to be direct manipulation: add from palette, remove from the barbell.

March 26, 20265 min read

Design Apps Build Logs

Numbers were not enough

I started with a classic utility mindset: show numbers, expose controls, keep everything explicit. That worked, but it could feel dense. The emotional moment came when I realized I was reading too much on my own screen before taking action. In a gym context, that is a warning sign. The interface should be understood in seconds, not parsed like a dashboard.

The concrete change was prioritizing the interactive barbell and tightening the add-remove loop. I kept symmetric behavior strict, made plate interactions direct, and verified with screen tests that adding a plate pair changes total weight and removing from the loaded bar returns it. That gave me confidence that the visual interaction was not just decorative. It was a functional control surface.

Glanceability became the rule

I leaned into visual hierarchy: bar first, totals visible, setup controls secondary. This keeps the cognitive path short. I want someone to glance, tap, and move. The builder mode is strongest when it feels like handling equipment, not filling forms.

The current working tree still suggests active UI refinement around the builder and visualizer. That feels right. This is where the app earns or loses trust in real use.

What I learned

The lesson is practical: in high-focus physical environments, clarity beats completeness. Every extra decision point costs attention, and attention is limited. I still need settings and profile control, but those should not dominate the first screen.

I want the main screen to feel obvious without losing precision. That is the design bar now.

Next move: run a first-time-user walkthrough test and remove any element that does not help someone load a bar faster.
