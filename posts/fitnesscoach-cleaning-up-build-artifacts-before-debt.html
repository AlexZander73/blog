FitnessCoach: Cleaning Up Build Artifacts Before They Became Debt

One of the most useful commits was not a feature commit. I removed generated artifacts and tightened build expectations before they quietly became team-wide friction.

March 21, 20263 min read

Build Logs Notes Systems

A quiet but strategic cleanup

The 2026-03-12 cleanup commit d03467a is easy to overlook, but it was one of the healthiest changes in this repository. I removed generated JS files, local workspace artifacts, and other non-source noise. I felt a little embarrassed when I saw how much generated output had slipped into tracking, because that usually signals blurred boundaries between source and build output.

Removing noise before it became normal

The concrete change was large but mechanical: dozens of generated files removed, with .gitignore tightened so they do not drift back in the same way. That included generated client JS duplicates and local workspace state artifacts that have no business in long-term history.

The immediate payoff was clarity. Diffs got smaller and more intentional. Reviewing actual product changes became faster because I was no longer parsing build churn. It also reduced accidental "works on my machine" residue in commits, which is a subtle but real risk when moving quickly.

Hygiene is velocity work

The bigger lesson is that repository hygiene is product velocity work. It is tempting to treat it as optional cleanup for later, especially when features are moving. But when you defer hygiene, every future commit pays tax. I would rather pay that cost once and keep momentum clean.

This commit also reinforced my preference for explicit build scripts and source-only commits. The less ambiguity there is about what should be versioned, the easier it is to move fast without creating hidden maintenance drag.

Next move: add CI guardrails that fail on committed generated artifacts so this problem does not reappear during rapid iteration.
