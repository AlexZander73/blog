FitnessCoach: Why I Pivoted to a Capacitor Client

I changed the client stack early, even though it meant touching almost everything at once. That decision hurt in the short term, but it gave the project a cleaner path to Android without a full rebuild.

March 18, 20264 min read

Apps Build Logs Systems

Paying for the hard decision early

I made the hardest architectural move early: I migrated the client direction to a Capacitor web app and removed the Expo-first path in commit e46f0c2 on 2026-03-12. That one commit touched the client, API, admin panel, shared packages, docs, and seed content. It was not elegant in the moment. I remember feeling that mix of relief and panic that happens when a decision is clearly right but still expensive.

One shared client, one clearer direction

The concrete change was simple to describe and hard to execute: one shared client codebase with React, Vite, TypeScript, and Capacitor, backed by a shared rules engine and shared contracts. I also documented the architecture immediately so the repo would not become a pile of disconnected decisions. That architecture summary became my anchor when the code got noisy.

The tradeoff was obvious. I accepted webview constraints and plugin complexity because I wanted platform continuity more than native-per-screen custom work. If I had stayed split across stack assumptions, I would have paid that cost later with more breakage and more migration debt. I wanted the pain once, not repeatedly.

What the pivot clarified

What I learned is that a cross-platform promise is only real when the core logic is platform-agnostic and the contracts are shared. A monorepo alone does not guarantee this. The package boundaries had to enforce it: rules in one place, contracts in one place, content in one place.

I still treat this as an early-stage build, not a finished product. But this pivot removed one major strategic uncertainty for me. I now know where this project stands when I think about iPhone-first today and Android expansion tomorrow.

Next move: lock plugin boundaries and document exactly which native capabilities will require custom bridge work.
