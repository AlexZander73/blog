Carpentry Companion: Packaging the App Into Plans

Carpentry Companion entry for 2026-02-27. I grouped core flows into packages (first called Work Packs) and introduced job limits for the free tier to make the value model understandable.

February 27, 20264 min read

Build Logs Tools Systems

From feature list to guided structure

Before this update, users could find tools, but the path through them was not clear. I grouped related flows into packages so roofing, stairs, and other tasks felt guided instead of scattered.

I also added free-tier job limits. That was not only monetization; it was a way to keep the app simple for first-time users while reserving deeper workflow volume for paid usage.

What I learned about naming

I first named these bundles "Work Packs." It was accurate, but the term felt heavy in UI. I kept shipping anyway because structure mattered more than perfect naming in that moment.

Emotionally, this day was about restraint. I wanted to add more, but the better move was to help users finish one task cleanly.

Lesson and next move

Lesson: users do not need more options. They need clearer paths.

Next step was native readiness. If this app was going to be used on real jobs, it needed reliable device behavior and a clean store pipeline.

Next move: move from web comfort to native reliability.
