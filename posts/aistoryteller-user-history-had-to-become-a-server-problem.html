AIStoryTeller: User History Had to Become a Server Problem

Personal story tools break the moment they blur one user into another. On 2026-02-20 AIStoryTeller drew that boundary at the server layer.

February 20, 20262 min read

Systems Apps Notes

Trust starts with boundaries

I think persistence bugs are product bugs before they are storage bugs. That is why the 2026-02-20 commit, Use server history per user, matters more than its size suggests. In a personal storytelling tool, mixed history is not a small flaw. It cuts straight into trust. If the wrong story, state, or memory shows up under the wrong person, the product stops feeling personal and starts feeling careless.

The fix here was clear in direction: move history separation to the server side. That matters because it changes where responsibility lives. Browser state is convenient, but convenience is not the same thing as ownership. Once AIStoryTeller began growing memory, world state, starters, and session behavior, user separation had to become explicit. A server boundary is not automatically perfect, but it is at least a serious answer to a serious problem.

When experimental becomes personal

The emotional moment I attach to this kind of change is not panic. It is a colder feeling than that. It is the moment I realize a system has crossed from experimental toy behavior into real user expectation. That is when I stop asking whether a feature technically works and start asking whether it protects the person using it. The concrete change was server-managed per-user history. The deeper change was treating identity as product state.

What I learned

The lesson is blunt: if a tool stores personal creative work, multi-user separation is not polish. It belongs in the core design. I can accept missing features longer than I can accept fuzzy ownership boundaries. This repo's history suggests AIStoryTeller learned that early enough to course-correct.

There is also a secondary lesson here about local-first apps. Local does not remove the need for clean boundaries. It just changes where those boundaries live.

Next move: keep all durable story state aligned to user and session identity before adding more adaptive behavior on top.
